Git in idea 入门操作

Posted on 2019-07-31 | In 工具 |
V1.0 by youngspring

一. 概念解释

  1. 概念

本地仓库（Local）：此次修改已经被提交，处于本地机器。但是其他同事还访问不到
远程仓库 (Remote)：代码在服务器上放置的位置，其他有权限的同事可以通过pull获取到

带origin的就是远程的
Commit:提交到本地仓库,还在你的机器上
Push:将本地仓库内容提交到远程仓库，这样大家都就可以访问到了。
合并代码：指的是将目标分支的代码合并到当前分支
代码版本：代码状态的唯一识别码

  1. 文件的几种状态

红色：本地文件，且未纳入到git的管理，git不会对其进行版本控制
这类文件加入到git管理中:
idea操作：

命令行操作：
git add 未纳入管理.md
绿色：已经纳入到了git的版本控制，但是是新建的，远程没有对应的代码
蓝色：被修改了的代码，待提交

  1. 各个按钮的作用

    回滚代码不建议轻易使用，这样会使你没有提交的工作都付之东流且不能找到。
  2. 文件夹结构

    .git
    保存了这个仓库的配置内容，一般不要动
    .gitignore

    git在进行版本管理的时候，是要知道哪些是要管理的，哪些是不要管理的，这个文件就是指明了这个。
    这个文件是不可或缺的。

二. 具体操作

  1. 初级操作
    1. 查看代码是谁提交的？
    2. 查看都谁改过这个文件
    3. 看当前代码都在哪些分支
  2. 复杂点的操作
    1. 合并代码
      先切换到当前分支(代码要合并进的分支)，pull,拉取最新代码

      进行合并
      远程分支的合并和当前分支的合并：
      见名知意，合并远程分支，就是将远程服务器的代码合并到当前的分支
      合并当前分支就是将本地仓库该分支的代码合并到当前分支
      建议大家在不明确需求的情况下不要擅自对分支进行合并操作

出现代码冲突的情况：

不要选择Accept Theirs 或者Accept Yours ,前者会冲掉自己的代码，后者你一提交会冲掉远程同事的代码，应该选择选择Merge

为什么会出现代码冲突？
两个不通的分支（本地、远程也算不同）在同一个文件的同一行上进行了修改操作，git不知道该选哪个

  1. 暂存代码


    将改动暂时存了起来，回头再来开发

  2. 将别的分支的代码合并进来Cherry-pick

    1. 高级的操作
      代码回滚(谨慎操作)

      回滚到那个指定阶段
      这时因为本地操作导致了本地分支落后于远程分支，所以需要强制提交
      Git push -f
      具体内容参见更详细的文档,
      在我们的项目中进行回滚操作是有很大风险的，极易造成代码丢失，所以禁止进行代码强制推送操作
  1. 常见问题：
    1. 当你代码已经提交，但是远程服务器没有，这时候你需要检查一下是不是知识提交到了本地仓库，而没有推送到远程服务器，这时，你需要

      向远程推送一下。当推送失败时，看下远程分支是不是被锁了，没有了权限
    2. 明明有了改动，但是idea却

      这时候，需要你
    3. 进行点无关紧要的改动，然后提交
    4. 命令行
      git status

      看下push里边有没有东西，然后push。这种情况一般常出现在合并代码后

三．好的使用习惯

  1. 习惯
    1. 在push代码前先pull一下
      这么做的原因：当你在一个文件上开发代码的时候，可能别人也在该文件上开发代码，很可能在将来合并代码的时候出现代码的冲突。当你push代码的时候，git会对比你的代码和远程（origin）服务器上的代码，如果有冲突，则会强制要求你先merge再进行提交。
      如果你不进行pull,直接push,这次merge就相当于是你意料之外的，会很容易处理错误。当你merge错误，再提交代码之后，就会将其同事的代码覆盖掉，引发不必要的麻烦
    2. 写代码前第一件事：pull代码，看看有没有其他同事提交代码
    3. 下班前第一件事：push代码，将今天的工作成果提交到服务器。安全，保证的代码不会丢失（即使电脑坏了），第一时间反馈给了其他在同一分支、文件上开发的同事，让他们第一时间处理冲突，这时候因为冲突代码量小，更容易处理，即使出错代价也是最小

2.快捷键
拉取代码的快捷键设置
File–>Settings–>Keymap–>Main menu–>VCS–>UPdate Project

SQL中in和exist的区别

Posted on 2019-07-30 |

1. 结论

  • in()适合B表比A表数据小的情况
  • exists()适合B表比A表数据大的情况
  • 当A表数据与B表数据一样大时,in与exists效率差不多,可任选一个使用.

2. 原理:

1. in()

select * from A where id in(select id from B)
等价于
SELECT id FROM B ----->先执行in中的查询
SELECT *  FROM A  WHERE A.id = B.id
  • 执行流程
    以上查询使用了in语句,in()只执行一次,它@1查出B表中的所有id字段并缓存起来.之后,@2检查A表的id是否与B表中的id相等,如果相等则将A表的记录加入结果集中,直到遍历完A表的所有记录.
    类似流程.
    每一个A的结果,都会遍历一次B的所有数据 
//查询过程类似于下边的代码
List resultSet=[];
Array A=(select * from A);
Array B=(select id from B);

for(int i=0;i<A.length;i++) {
   for(int j=0;j<B.length;j++) {
      if(A[i].id==B[j].id) {
         resultSet.add(A[i]);
         break;
      }
   }
}
return resultSet;

  1. A表中有100条记录，B表中有1000条记录，那么最多可能遍历100*1000次，效率很差

  2. A表中有1000条记录，B表中有100条记录，那么最多可遍历1000*100此，内循环次数减少，效率大大提升

    看似遍历数据次数相同,但是由于遍历B的更多的是无用遍历,B结果集越大,命中率就越低,所以减少内循环次数可以提高效率

2. exist()

SELECT  字段 FROM  table WHERE EXISTS（subquery）;
SELECT * FROM A WHERE EXISTS(SELECT 1 FROM B WHERE B.id  = A.id);
 等价于
SELECT * FROM A;
SELECT I FROM B WHERE B.id = A.id;
  • 执行流程:
    EXISTS()查询会执行SELECT * FROM A查询，执行A.length次，并不会将EXISTS()查询结果结果进行缓存，因为EXISTS()查询返回一个布尔值true或flase，它只在乎EXISTS()的查询中是否有记录，与具体的结果集无关。
    EXISTS()查询是将主查询的结果集放到子查询中做验证，根据验证结果是true或false来决定主查询数据结果是否得以保存。

    如果A的查询结果在B查询中返回了true,则保存结果集 

//查询过程类似于下边的代码
List resultSet=[];
Array A=(select * from A)

for(int i=0;i<A.length;i++) {
   if(exists(A[i].id) {    //执行select 1 from B b where b.id=a.id是否有记录返回
       resultSet.add(A[i]);
   }
}
return resultSet;

每有一条A数据都会执行一次Exist()

  1. 如:A表有10000条记录,B表有1000000条记录,那么exists()会执行10000次去判断A表中的id是否与B表中的id相等.
  2. A表有10000条记录,B表有100000000条记录,那么exists()还是执行10000次,因为它只执行A.length次,可见B表数据越多,越适合exists()发挥效果.
  3. A表有10000条记录,B表有100条记录,那么exists()还是执行10000次,还不如使用in()遍历10000*100次,因为in()是在内存里遍历比较,而exists()需要查询数据库,我们都知道查询数据库所消耗的性能更高,而内存比较很快.
